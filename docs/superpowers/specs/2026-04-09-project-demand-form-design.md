# 横向项目需求表单设计说明

## 背景

- 当前仓库已经具备登录、首页和基础布局能力，但还没有横向项目需求填报模块。
- 目标页面来自一张传统审批系统表单截图，字段密集、信息量大，并包含顶部流程操作区、主体表单区、附件区和小组成员表格区。
- 本次范围已经确认：先完成页面结构、现代化布局、本地表单校验以及小组成员增删改，不接真实提交接口。

## 目标

- 新建独立的 `project-demand` 业务模块，作为后续“新建、编辑、查看流程、详情”扩展的基础。
- 按参考图还原核心表单结构，但视觉实现采用当前项目的 `Naive UI + Tailwind CSS` 风格，而不是复刻旧系统样式。
- 页面仅面向浏览器 Web 端，不考虑移动端适配，但需要兼顾不同桌面屏幕下的字号层级与阅读舒适度。
- 表单必须具备本地必填校验、基础格式校验、提交与暂存两类操作反馈。
- 小组成员区域必须支持新增、编辑、删除，并具备最小必填校验。

## 非目标

- 本阶段不接真实后端接口、不实现真实上传、不实现真实流程查看。
- 本阶段不做复杂审批权限判断、流程状态回显、历史版本管理。
- 本阶段不新增 Pinia store，页面状态仅保留在模块 composable 内。
- 本阶段不修改 `src/route-map.d.ts`。

## 约束与决策

### 模块边界决策

- 采用新模块 `src/modules/project-demand/`，不放入 `dashboard` 等现有模块中。
- 遵循项目现有模块约束：仅使用 `pages / components / composables` 三层结构。
- 页面入口保持轻量，表单状态、成员数据、校验逻辑统一放在 composable 中管理。
- 通过文件路由提供业务页 `project-demand`，保证点击“横向项目”时可以跳转到对应业务路由。

### UI 决策

- 页面骨架、分栏、间距和响应式布局使用 Tailwind CSS。
- 输入框、日期、选择器、按钮、消息提示、表格和弹层优先使用 Naive UI。
- 保留参考图的字段组织和业务分区，但采用当前项目更现代、更清晰的卡片式页面结构。
- 本阶段不为手机端布局做专门适配，设计基线以桌面浏览器为主。
- 字体和间距需要在常见桌面分辨率下保持稳定层级，避免在大屏或较窄浏览器窗口中出现字号失衡。

### 数据策略决策

- 使用本地默认值和本地选项集合，不提前绑定真实接口模型。
- `useProjectDemandPage.ts` 作为页面状态唯一来源，统一暴露表单值、下拉选项、成员列表和操作方法。
- “提交”执行完整校验并给出前端反馈；“暂存”只做轻量处理，不阻断未完成表单。

## 模块边界

### 页面

- `src/modules/project-demand/pages/index.vue`
  - 作为文件路由入口页。
  - 只负责组装工具栏、主体表单和小组成员区域。
  - 不直接承载复杂校验、默认值映射或成员数据操作。

### 组件

- `src/modules/project-demand/components/ProjectDemandToolbar.vue`
  - 承载顶部流程相关控件和主操作按钮。
  - 通过 `props / emits` 接收字段值并向外抛出提交、暂存、返回、查看流程等事件。

- `src/modules/project-demand/components/ProjectDemandForm.vue`
  - 承载主体字段网格、备注区、合同模板区和附件入口区。
  - 负责展示字段、错误提示和局部交互，不直接管理业务状态。

- `src/modules/project-demand/components/ProjectDemandMemberTable.vue`
  - 承载小组成员列表和“新增 / 编辑 / 删除”交互。
  - 使用表格呈现成员信息，并通过事件把修改意图上抛给页面。

### Composable

- `src/modules/project-demand/composables/useProjectDemandPage.ts`
  - 管理整个页面的响应式状态。
  - 提供顶部选项列表、主体表单默认值、成员列表默认值。
  - 封装提交、暂存、成员增删改、校验和消息提示。

## 组件图与职责

- `ProjectDemandPage`
  - 负责页面组装和组件通信。
- `ProjectDemandToolbar`
  - 单一职责：流程配置与顶层动作触发。
- `ProjectDemandForm`
  - 单一职责：展示并编辑主体业务字段。
- `ProjectDemandMemberTable`
  - 单一职责：展示并维护成员列表。

组件通信约定：

- 页面将统一状态拆分为 `toolbarModel`、`formModel`、`memberRows` 传给子组件。
- 子组件只通过显式 `emits` 回传字段更新与用户动作。
- 不使用 provide/inject，不在子组件中直接依赖 composable。

## 页面结构设计

### 整体结构

- 页面采用纵向三段式：顶部工具栏卡片、主体表单卡片、小组成员卡片。
- 每个区块均使用白底容器、浅色边框、紧凑间距，与当前首页页面风格保持一致。
- 页面内容最大宽度保持充足，避免字段区过窄导致桌面端频繁换行。
- 页面默认服务于桌面浏览器宽度区间，允许在较窄窗口中自然压缩，但不以移动端单列方案作为验收目标。

### 顶部工具栏

- 放置以下字段与操作：
  - 是否为中标项目
  - 选择流程
  - 办理人
  - 微信提醒
  - 提交
  - 暂存
  - 返回
  - 查看流程
- 在桌面端采用多列横排；在较窄浏览器窗口下允许自然换行，但不单独设计移动端工具栏布局。
- 主按钮使用项目主色，次按钮使用轻量边框或次级色。

### 主体表单区

- 使用面向桌面的弹性网格组织字段：
  - 大屏优先双列布局，每列内部字段纵向排列。
  - 在较窄桌面窗口下允许退化为更宽松的单列或混合布局，但不以移动端单列适配为目标。
- 字段标签保持统一宽度或统一对齐方式，减少视觉噪音。
- 备注区、合同模板区、附件区跨整行展示。

### 小组成员区

- 顶部展示标题和“添加”按钮。
- 主体使用 Naive UI 表格承载“名次、姓名、工作证号、电话、身份、操作”。
- 当没有成员时展示空态文案，而不是渲染空白表格。

## 交互规则

### 表单交互

- 页面加载时填入一组演示默认值，使布局和控件状态可直接观察。
- 选择器、日期和输入框都绑定到 composable 的统一状态对象。
- `提交` 时执行完整校验，通过后只展示“提交成功（模拟）”反馈。
- `暂存` 时仅提示“已暂存（模拟）”，不强制所有必填项完整。
- `返回` 首版只执行路由返回或回首页逻辑，具体去向在实现阶段按当前路由结构决定。
- `查看流程` 首版只提示“流程预览待接入”。

### 成员交互

- 初始列表带一条示例成员数据。
- 点击“添加”打开轻量弹层表单，录入后追加到列表尾部。
- 点击“修改”打开同一弹层并回填当前成员数据。
- 点击“删除”前给出二次确认。
- 名次默认可自动递增，但允许用户修改。

## 校验设计

### 主体表单校验

- 参考图中带星号的字段优先列为必填项，至少包括：
  - 项目名称
  - 开始时间
  - 所属单位
  - 委托单位
  - 委托单位单位类型
  - 委托单位所在省
  - 委托单位所在市
  - 中期节点
  - 校区
  - 经办人联系电话
- 金额字段校验为非负数字。
- 电话字段做基础手机号或座机格式校验，首版只做宽松校验即可。
- 日期字段只校验是否为空，不做复杂前后关系限制，除非后续需求明确。

### 成员校验

- 新增或编辑成员时，至少校验：姓名、工作证号、身份。
- 电话字段可选填，填写时校验格式。
- 同一列表内工作证号建议避免重复，重复时给出提示。

## Web 端布局策略

- `xl` 及以上：顶部工具栏横向布局，主体表单双列网格，成员区全宽表格。
- `lg` 到 `xl`：顶部控件允许换行，主体表单保持双列但放宽字段间距。
- `md` 到 `lg`：可退化为单列或混合布局，优先保证字段可读性和操作区完整性。
- 表格在较窄桌面窗口下允许横向滚动，避免强行压缩列宽。
- 标题、字段标签、正文和辅助信息要建立明确字号层级，并在不同桌面宽度下保持稳定观感。

## 状态与数据流

### 状态来源

- `reactive` 管理主表单对象和顶部工具栏对象。
- `shallowRef` 管理加载态、弹层开关、当前编辑成员索引、提交中状态和提示文案。
- 成员列表使用 `ref` 或 `reactive` 数组维护，所有衍生显示数据通过 `computed` 提供。

### 数据流

1. 页面调用 `useProjectDemandPage()`。
2. composable 返回统一状态、下拉选项、成员操作方法和提交方法。
3. 页面把状态拆分传给三个子组件。
4. 子组件通过字段更新事件和动作事件回传到页面。
5. 页面调用 composable 方法修改状态。
6. 成功或失败反馈通过 Naive UI 消息组件呈现。

## 测试策略

- 重点测试 composable，而不是为页面样式写大量快照测试。
- 核心测试范围：
  - 初始化默认值是否正确。
  - 提交时必填校验是否生效。
  - 暂存是否允许未完成表单。
  - 成员新增、编辑、删除是否正确更新列表。
  - 成员重复工作证号校验是否生效。
- 页面层至少补一条基础渲染测试，确保关键区域标题存在。

## 验收标准

- 可以通过独立业务路由 `/project-demand` 进入横向项目需求页面，且“横向项目”入口能正确跳转。
- 页面包含顶部流程操作区、主体表单区、备注与附件区、小组成员区。
- 页面采用 Naive UI + Tailwind 的现代化布局，而不是旧式表格拼接视觉。
- 点击提交时能看到明确校验提示。
- 点击暂存时能完成本地保存反馈而不强制完整校验。
- 小组成员支持新增、编辑、删除。
- 页面在不同桌面屏幕和浏览器宽度下保持清晰的字号层级与可读性。

## 实现注意事项

- `.vue` 页面和组件保持轻量，复杂状态编排集中到 composable。
- 不新增模块内 `api/`、`store/`、`types/` 目录。
- 不直接修改 `src/route-map.d.ts`，由文件路由自动生成机制处理。
- 完成实现后必须执行 `pnpm check`。
